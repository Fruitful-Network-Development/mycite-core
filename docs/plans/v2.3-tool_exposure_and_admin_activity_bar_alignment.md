# V2.3 Tool Exposure And Admin Activity Bar Alignment

Authority: [v2-authority_stack.md](v2-authority_stack.md)

## Summary

Current V2 already has a working admin tool registry and live deployed
tool-bearing shell on `/portal/system`, but it does not yet have an
instance-level V2 gate for visibility. This plan adds that missing layer without
reverting to the V1 config-driven tool model.

The forward rule is:

- shell registry stays canonical for legality
- `private/config.json.tool_exposure` becomes the new visibility gate
- the gate is `hide-and-block`
- admin-first `/portal/system` is the first surface that uses it

## Current findings that shape the implementation

- Current code already exposes `aws`, `aws_narrow_write`, `aws_csm_sandbox`,
  and `aws_csm_onboarding` through the live admin shell.
- Current V2 does not read `private/config.json` for activity-bar visibility.
- Legacy `tools_configuration` still exists in state and docs, but it is not the
  correct V2 source of truth.
- AWS and AWS-CSM are the only positive V2 reference family for how a tool is
  created, cataloged, exposed, and tested.

## Required implementation changes

### Host config and parsing

- extend the V2 host config loader to read optional `tool_exposure`
- treat missing keys as disabled
- keep config parsing separate from shell registry construction

### Shell and runtime behavior

- preserve the existing shell-owned registry order and audience rules
- filter visible admin tools by dual gate:
  - shell registry says the tool exists and is legal
  - `tool_exposure.<tool_id>.enabled` is true
- enforce the same rule for launch, not only for display
- surface the filtered tool set in admin status output so operators can debug
  instance config quickly

### Interface surface behavior

- admin activity bar on `/portal/system` shows only filtered tools
- tool labels, ordering, and routing still come from the shell registry
- trusted-tenant portal slices remain outside this plan

### Operational and documentation work

- document the new `tool_exposure` shape in repo contracts and live-state docs
- add default examples for FND and TFF configs only after runtime support lands
- update stale post-AWS catalog docs so they reflect current admin tool reality

## Sequencing

1. Implement the `tool_exposure` contract and admin-shell filtering.
2. Add tests for hidden-and-blocked behavior.
3. Update live FND and TFF configs with explicit `tool_exposure` once code lands.
4. Reopen individual tool plans only after the gate exists.

## Test plan

- unit tests for config parsing and missing-key defaults
- unit tests for dual-gate selection and launch denial
- integration tests for `/portal/api/v2/admin/shell` filtered tool output
- integration tests proving disabled tools cannot be launched directly
- architecture tests proving config does not become the legality source
- host/live smoke checks proving FND and TFF reflect config-enabled tool sets

## Assumptions

- `/portal/system` remains the first tool-bearing V2 surface
- `tool_exposure` is a new V2 section, not a rename of `tools_configuration`
- trusted-tenant tool exposure is a later decision and is not bundled into this
  admin-first pass
- existing current V2 AWS tools remain the reference implementation family
