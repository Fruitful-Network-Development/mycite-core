# V2.3 IO Surface And Datum Handling Alignment

Authority: [v2-authority_stack.md](v2-authority_stack.md)

## Audit verdict

The repo is only partially aligned for datum handling and stable interface
surfaces.

What was already aligned:

- shell ownership remains in `packages/state_machine/`
- trusted-tenant and admin runtime envelopes are stable
- tool legality is shell-owned and catalog-driven
- repo test coverage for runtime/host boundaries is strong

What was not aligned before this plan:

- datum handling stopped at anthology/publication projections plus simple
  datum-ref parsing
- active instance datum truth still lived under `data/sandbox/*/sources`, but V2
  did not mediate those documents
- legal anchor references such as `rf.3-1-3` were not recognized separately from
  content/family problems
- illegal placeholders such as `HERE` were not preserved-and-flagged through a
  stable V2 datum surface
- datum readiness collapsed anthology availability and derived materialization
  into one weak signal
- trusted-tenant shell-region contracts were not documented separately from the
  admin shell contract

Deployed portal drift remains a separate problem and is intentionally split into
the follow-on plan:

- [v2.3-deploy_live_surface_stability.md](v2.3-deploy_live_surface_stability.md)

## Implemented repo alignment

### Ownership preserved

- `packages/state_machine/` remains unchanged.
- `private/config.json` remains exposure metadata only.
- `data/payloads/*.bin`, `data/payloads/cache/*.json`, `data/resources/*`, and
  `data/references/*` remain derived or compatibility surfaces, not canonical
  datum truth.

### Authoritative datum read seam

`packages/ports/datum_store/` now exposes a read-only authoritative datum
document seam for:

- `data/system/anthology.json`
- `data/sandbox/<tool>/sources/*.json`

The filesystem adapter reads those documents without rewriting raw tuples or
magnitudes. Supporting sandbox anchor files are loaded only to resolve legal
`rf.*` abstraction references for recognition.

### Read-only recognition and diagnostics

`packages/modules/domains/datum_recognition/` now projects authoritative datum
documents into:

- raw row preservation
- normalized legal `rf.*` reference bindings
- recognized family/anchor when resolvable
- row diagnostics:
  - `ok`
  - `missing_reference`
  - `unresolved_anchor`
  - `family_magnitude_mismatch`
  - `illegal_magnitude_literal`
  - `address_irregularity`
  - `unrecognized_family`
- render hints for presentation-only overlays/lenses

This means V2 now keeps values like `HERE` intact while flagging them as
diagnostic states instead of rejecting or rewriting them.

### Surface alignment

The existing admin datum surface remains the single datum shell path.

- `datum.resource_workbench` still owns the datum workbench slice.
- The admin runtime now composes that slice from authoritative datum documents
  plus read-only recognition diagnostics.
- The shell inspector now has an explicit datum-summary region contract instead
  of leaving datum semantics to ad hoc JSON inspection.
- The direct host route `/portal/api/v2/data/system/resource-workbench` now
  returns the same recognized datum workbench projection rather than the old
  anthology-only summary.

### Readiness tightening

Datum readiness now distinguishes:

- authoritative anthology availability
- authoritative catalog availability
- sandbox source-document presence
- derived materialization completeness

Host healthz exposes those states without reclassifying derived payload/cache
artifacts as source truth.

## Completion gate

This plan is complete when:

- repo contracts expose one authoritative datum-document read seam
- runtime/admin datum surfaces consume one read-only recognition path
- legal anchor references are distinguished from family/content problems
- illegal literals such as `HERE` are preserved and flagged
- readiness semantics separate authoritative availability from derived
  materialization completeness
- tests cover port, adapter, unit, integration, and architecture boundaries

Full datum mutation, safe repair, and deploy/live rollout are explicitly out of
scope for this plan.
