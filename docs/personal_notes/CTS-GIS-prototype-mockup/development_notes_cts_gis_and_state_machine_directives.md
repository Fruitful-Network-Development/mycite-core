## Gap Assessment and Recommendations for CTS‑GIS SAMRAS Integration

### 1 Overview of the current design

The CTS‑GIS tool uses **SAMRAS** (Shape‑Addressed Mixed‑Radix Address Space) as the structural authority for its namespace. The contract documents in the repository explain that SAMRAS stores **child counts** rather than addresses, decodes them breadth‑first, and derives node addresses from ordinal position[source](https://github.com/Fruitful-Network-Development/mycite-core/blob/main/docs/contracts/samras_structural_model.md#L9-L59). The UI should treat **administrative node rows as overlays** that decorate the derived structure; the SAMRAS engine remains the sole authority for hierarchy shape[source](https://github.com/Fruitful-Network-Development/mycite-core/blob/main/docs/contracts/cts_gis_samras_addressing.md#L9-L20). When the underlying magnitude is invalid, the system must fail closed and display diagnostics rather than generating a new hierarchy from row data[source](https://github.com/Fruitful-Network-Development/mycite-core/blob/main/docs/contracts/cts_gis_samras_addressing.md#L42-L49). The audit file notes that drift occurred when CTS‑GIS navigational logic began constructing the tree from administrative rows; this led to duplicate node bindings, invalid parent/child relationships, misplaced branches, and incomplete titles[source](https://github.com/Fruitful-Network-Development/mycite-core/blob/main/docs/audits/cts_gis_samras_history_audit_2026-04-17.md#L66-L102). To remediate this, current policies emphasise re‑enabling the SAMRAS engine and treating row data as non‑authoritative overlays[source](https://github.com/Fruitful-Network-Development/mycite-core/blob/main/docs/audits/cts_gis_samras_history_audit_2026-04-17.md#L105-L112).

The new plan introduces **SAMRAS context values** like `attention`, `intention` and `time` to the CTS‑GIS state machine. It also proposes a new **Garland section** that overlays the geospatial projection on the profile projection, giving visual control over precinct boundaries. New data files under `sandbox/cts‑gis/sources/precincts` are intended to optimise source loading and reduce portal overhead.

### 2 Findings from repository documentation

1. **SAMRAS structural model** – The `samras_structural_model.md` document defines the canonical encoding (address‑width field, stop‑count field, stop address array and value stream). It emphasises that derived addresses are contiguous and breadth‑first and that canonical writes persist only the bitstream[source](https://github.com/Fruitful-Network-Development/mycite-core/blob/main/docs/contracts/samras_structural_model.md#L25-L61). This underscores that the UI should not edit raw magnitudes directly and that structure must remain engine‑driven[source](https://github.com/Fruitful-Network-Development/mycite-core/blob/main/docs/contracts/samras_engine_ui_boundary.md#L9-L27).
2. **CTS‑GIS SAMRAS addressing** – The `cts_gis_samras_addressing.md` contract states that CTS‑GIS resolves the SAMRAS structure by scanning caches and tool files for decodable magnitudes, then falls back to administrative rows only when necessary[source](https://github.com/Fruitful-Network-Development/mycite-core/blob/main/docs/contracts/cts_gis_samras_addressing.md#L13-L21). It describes how Garland couples to the active SAMRAS node: the profile projection may be blank when no profile exists, while the geospatial projection remains empty until projectable geometry appears[source](https://github.com/Fruitful-Network-Development/mycite-core/blob/main/docs/contracts/cts_gis_samras_addressing.md#L132-L140). The document also specifies selection normalisation rules for `attention` and `intention` values[source](https://github.com/Fruitful-Network-Development/mycite-core/blob/main/docs/contracts/cts_gis_samras_addressing.md#L122-L127).
3. **Engine/UI boundary** – `samras_engine_ui_boundary.md` clarifies that the engine owns decode/encode, validation and authority selection, while the UI owns tree inspection, dropdown presentation and user intentions[source](https://github.com/Fruitful-Network-Development/mycite-core/blob/main/docs/contracts/samras_engine_ui_boundary.md#L9-L27). This separation must be respected when adding new features like context‑driven state machines or Garland overlays.
4. **Validity and mutation** – `samras_validity_and_mutation.md` lists conditions for structural validity (e.g., contiguous roots, strictly increasing stop addresses, complete value stream) and describes canonical mutation flows[source](https://github.com/Fruitful-Network-Development/mycite-core/blob/main/docs/contracts/samras_validity_and_mutation.md#L9-L34). Any new SAMRAS magnitude (e.g., for precinct sets) must satisfy these conditions.
5. **Historical audit** – The audit dated 2026‑04‑17 shows that current live data contains multiple defects: invalid magnitudes in the cache, duplicate node bindings, mismatched parent/child relationships and misplaced root branches[source](https://github.com/Fruitful-Network-Development/mycite-core/blob/main/docs/audits/cts_gis_samras_history_audit_2026-04-17.md#L66-L96). These issues underscore the importance of using a valid SAMRAS authority and verifying that administrative rows align with derived addresses before they are overlaid.

Naming conventions should follow the SAMRAS node id (e.g., `3‑2‑3‑17‑77‑1‑6‑4‑1‑4.json`) to allow the engine to locate the file based on the current node context.

### 7 Garland and UI redesign considerations

The plan proposes significant changes to the Garland section:

1. **Resize the Garland vs. Diktataograph sections** – making Garland occupy three‑fifths of the window and shrinking the Diktataograph section would prioritise geospatial and profile information. This aligns with the contract that Garland is driven by the selected SAMRAS node and should display profile and geospatial projections[source](https://github.com/Fruitful-Network-Development/mycite-core/blob/main/docs/contracts/cts_gis_samras_addressing.md#L132-L140). However, resizing alone does not solve the underlying drift; the engine must still provide valid data.
2. **Overlay geospatial projection onto the profile projection** – layering the map over the profile view may help users see spatial context alongside profile details. Care must be taken to preserve interactive elements (e.g., toggles and summary objects) and to avoid obscuring profile text.
3. **Remove the “Current Profile” section** – instead of a static profile panel, the plan suggests using a summary object within the profile projection sub‑section to toggle visible layers (e.g., precinct boundaries, demographic overlays). This could make the interface more flexible. The summary object should indicate which layers are currently displayed and allow users to enable/disable them.
4. **Context‑driven state machine** – the proposed directive language `[<directive>;(<sandbox>.<file>.<datum>.<attention>.<intention>.<time>.<archetype>.<space>:target)]` introduces an expressive way to specify state transitions. While powerful, it may increase complexity for both developers and users. To ensure maintainability:

   - **Provide a parser and type system**: instead of treating the directive string as an opaque token, define a JSON or structured representation with fields for each context value. This improves validation and introspection.
   - **Clarify defaults**: the default state example uses `attention=3-2-3-17`, `intention=today~4-447-751-507-818`, and `archetype=SAMRAS_family` but does not explain how these values are derived. Documenting how the engine resolves unspecified fields would reduce confusion.
   - **Limit UI responsibilities**: recall that the UI does not own structural mutation or authority selection[source](https://github.com/Fruitful-Network-Development/mycite-core/blob/main/docs/contracts/samras_engine_ui_boundary.md#L9-L27). The directive engine should reside in the backend, and the UI should send high‑level intents rather than low‑level node addresses.
5. **Handling `time` context** – the plan states that the `time` context is used only when an information object uses abstracted HOPS chronological addresses. This suggests that not all profiles or precincts will have time‑based data. The state machine must therefore support a `null` or default time context and gracefully degrade when no time dimension is available.
