## OUTPUT

Context and goals
Fruitful Network Development’s (FND) websites are currently split between three places:
    1. Main branch static site – the clients/fruitfulnetworkdevelopment.com/frontend/ directory on the webDZ repository has plain HTML pages and CSS/JS assets (no manifest). For example, the index.html page contains a long, monolithic <section> structure for the hero, process sequence, service overview and editorial sections, as well as separate static HTML for pricing cards, technology call‑outs and footers[1]. All content is hard‑coded, making updates and editing difficult.
    2. save_point-04-15-2026 branch manifest – an experimental branch in webDZ where clients/fruitfulnetworkdevelopment.com/frontend/assets/docs/manifest.json stores the same content as index.html but placed inside a main_html string for each page. Although generated by render_manifest.py, it lacks the modular structure seen in other site manifests and mixes large HTML fragments with JSON metadata.
    3. Design considerations in mycite-core – the FND-DCM_considerations.txt document explains the vision for a site‑editor tool. It states that the manifest should be the canonical schema for navigation, page structure, content, images and icons, and that pages must include stable IDs and arrays for each section[2]. It also recommends adding an enabled flag to optional sections, editor metadata (label, type, visibility conditions) and using a draft–preview–publish workflow[3].[4]
Across the larger organization there are two other sites:
    • Trapp Family Farm (TFF) – uses webdz.site_content.v3 manifest. The manifest cleanly defines site‑level assets, navigation, icons, footers, collections and pages. Each page chooses a template (e.g., home_featured, section_stack), and the content is expressed as structured sections with id, heading, body arrays, lists of images, cards and subsections[5][6].
    • Cuyahoga Valley Countryside Conservancy (CVCC) – uses webdz.site_content.v2 manifest. It similarly defines site metadata, navigation, a lean footer, collections and pages with templates. Content like hero images, intro panels and mission blocks is stored in lists with IDs and headings[7][8].
The goal is to consolidate the purpose‑driven content from the FND save_point-04-15-2026 branch into a modular manifest like the TFF and CVCC examples, while preserving the public site on the main branch during the cut‑over. The consolidated structure should be flexible enough to support multiple sites (FND, TFF, CVCC and future clients) with editable icons, text, images, colours and fonts. It should also be compatible with the planned site‑editor tool so that non‑technical users can select fields in a preview, edit them and publish a new site version.[4]
Proposed canonical manifest schema
    1. Site metadata – A schema key (webdz.site_content.v3), name, homepage_href and description. A shell object should include header/footer options (menu label, wordmark, colours) and asset references for favicon, hero graphic, header/footer graphics. ̩These values come from the current FND branding; for example, the FND home page uses a blue/green/yellow palette, an FND wordmark and a hero graphic of the logo[9].
    2. Icons section – Consolidate icons used across the site into a mapping: navigation icons, footer icons and per‑section icons. In TFF’s manifest, icons are grouped by function (navigation, footer and sections)[10]; adopting this structure allows global icon selection and reuse. Use relative paths to the assets/icon/… directory.
    3. Navigation array – Each entry should include id, label, href and icon fields, matching the existing FND menu items (Home, Services, Research, About, Contact, Payment‑Processing/Checkout). TFF’s navigation array provides a template: each item has an ID, a label and an icon path[11]. The href should correspond to the target page’s file property.
    4. Footer – Use a footer object with columns listing contact details, copyright text and call‑to‑action sections. Each column can be templated (e.g., contact_lines, line_list, social_links) as in the TFF manifest[12]. Include FND’s contact details (founder contact info, social icons, subscribe form) as separate items[13].
    5. Collections – If articles or research docs (Markdown files) are to remain separate, define them as markdown_directory collections. TFF uses a newsletters collection to pull markdown documents[14]. FND’s research articles (e.g., market_transparency.md) and blog posts can be added as collections for dynamic pages like subpages/articles.html.
    6. Pages dictionary – Each key matches a page id (e.g., home, services, research, about, contact, payment_processing, checkout). Each page has:
    7. file – output HTML file (e.g., index.html).
    8. template – one of the standard templates (home_featured, section_stack, article_archive, static_fragment, etc.) or new templates defined for FND content (see below).
    9. title and description – used for <title> and <meta> tags.
    10. body_class (optional) – page‑specific CSS class.
    11. shell overrides (e.g., show_header_graphic, show_main_nav).
    12. content or sections – structured data used by the template. Use stable id fields and arrays for lists; add an enabled boolean to toggle optional blocks[3].

Editor metadata and flags
To support a site‑editor tool, embed additional keys in the manifest:
    • enabled flag – Add "enabled": true to every section, card or list item; editors can toggle these without deleting data[3].
    • id field – Ensure every section, card and item has a stable identifier; avoid using array indices in path references[17].
    • editor object – Provide a label and a data type for each field (e.g., type: text, type: rich_text, type: image, type: icon, type: link, type: boolean). Optional visible_if conditions allow dynamic UI. This metadata helps the inspector panel display appropriate controls[4].

Guidance for cross‑site modular design
    • Reuse templates: Define universal templates (hero, process sequence, section stack, rich cards, table sections, newsletter form) in a shared library within webDZ. TFF’s toc_section, split_section and subsection_collection illustrate how nested lists and images can be expressed consistently across pages[16]. CVCC’s archive_home and timeline_stack templates show how to display collections of items like board profiles and historical events[20].
    • Common icon sets: Place all icons under assets/icon/ and categorise them by function (UI icons, domain-specific icons, logos). Use the icons mapping in the manifest for navigation and per‑section icons so that editors can swap icons easily.
    • Colour and typography tokens: Use CSS variables for colours and fonts, and store the values in per‑site theme files. Ensure that templates refer to variables (e.g., var(--color-accent-blue)) instead of hard-coded colours. This allows each site to adopt a unique look while sharing the same HTML structure.
    • Stable IDs and enabled flags: Make sure every section and card has a human‑readable id and an enabled boolean. This prevents ID breakage when reordering sections and allows optional content to be hidden without deletion[3].
    • Editor metadata: Attach an editor object to each field specifying the type (text, rich_text, image, link, boolean) and label. Use conditional display (visible_if) when certain fields should only appear in the editor if a related flag is true (e.g., show price table only when pricing is enabled).[4]
    • Documentation and training: Write developer and editor documentation explaining how to add new templates and how to update the manifest. Encourage using Markdown collections for articles to keep long‑form content separate from the manifest, aligning with the house style guidelines for articles (reference tone, topic purity, heading structure, citations and length guidelines)[21][22].
