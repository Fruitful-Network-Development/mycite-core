Remove the current live CTS-GIS content implementation and strip the surface back to a frame-only scaffold. Do not preserve the existing Diktataograph content logic, Garland content logic, readiness-conditioned rendering rules, source-evidence-driven population, or any current attempt at dynamic hierarchy rendering. 1. context strip
2. Diktataograph pane

- the CTS-GIS frame always renders
- the four sections always exist
- the layout is stable and inspectable
- the content is synthetic and deterministic

- the four CTS-GIS frame regions render consistently

1. CTS-GIS renderer payload updated to support an explicit ordered-hierarchy mode
2. new dedicated ordered-hierarchy renderer function
3. static synthetic hierarchy rendered through that new mode
4. Garland still rendered with synthetic polygon and synthetic profile summary

6. a final controlled pass that swaps synthetic hierarchy nodes for real CTS-GIS node structures

## PLAN

> (Stages 1-4 in One Cycle)

**Summary**
- Rebuild CTS-GIS from a deterministic scaffold with four always-visible regions: `context strip`, `Diktataograph`, `Garland geospatial`, `Garland profile`.
- Replace branch-card Diktataograph as default with a separate ordered-hierarchy renderer mode using dynamic two-column `msn_id`/`title` blocks.
- Deliver in strict sequence inside one cycle: frame lock, synthetic content, ordered renderer, real Garland geometry swap, real hierarchy swap.
- Keep only minimal portal-shell integration; readiness/evidence states cannot suppress or reshape the four-region interface.

**Interface Contract Changes**
- Add `navigation_canvas.mode` to CTS-GIS interface payload.
- Set `navigation_canvas.mode = "ordered_hierarchy"` as default.
- Keep `"legacy_branch_canvas"` as alternate mode for the current branch-card renderer.
- Add `navigation_canvas.ordered_hierarchy` payload for dynamic block-chain rendering (`columns`, `active_path`, `selected_node_id`, interaction metadata).
- Keep legacy `anchored_path`, `structure_field`, and `projection_rule_field` only for legacy mode compatibility.
- Enforce title-column rule: if no ASCII-decodable title exists in `sc.3-2-3-17-77-1-6-4-1-4.msn-administrative.json`, render title blank (no ambiguous-label fallback).

**Implementation Changes**
- Refactor CTS-GIS runtime assembly in [portal_cts_gis_runtime.py](/srv/repo/mycite-core/MyCiteV2/instances/_shared/runtime/portal_cts_gis_runtime.py) so frame/layout, synthetic content, and real-data swaps are separate steps.
- Make scaffold regions unconditional in the interface body regardless of readiness, projection availability, or document presence.
- Build deterministic synthetic content first (fake lineage/children, fake polygon, fake profile summary) before any real mapping step.
- Add explicit real-wiring pass 1 for Garland geospatial: map real HOPS-derived geometry into `geospatial_projection.feature_collection`.
- Add explicit real-wiring pass 2 for Diktataograph: replace synthetic hierarchy with full SAMRAS-magnitude namespace expansion and ASCII title correlation.
- Add renderer-mode dispatch and new dedicated ordered renderer in [v2_portal_inspector_renderers.js](/srv/repo/mycite-core/MyCiteV2/instances/_shared/portal_host/static/v2_portal_inspector_renderers.js); keep `renderStructureCanvas` intact as legacy.
- Implement structural selection expansion/compression behavior and dynamic depth block chaining in ordered mode.
- Add ordered-hierarchy styling and responsive support in [portal.css](/srv/repo/mycite-core/MyCiteV2/instances/_shared/portal_host/static/portal.css), while retaining legacy classes needed for compatibility.
- Use `/srv/repo/mycite-core/docs/personal_notes/CTS-GIS-prototype-mockup/` as the locked historical interaction reference for ordered-hierarchy behavior.

**Test Plan**
- Runtime tests assert all four CTS-GIS regions always render with stable section titles in empty and no-data states.
- Runtime tests assert default `navigation_canvas.mode` is `ordered_hierarchy`.
- Runtime tests assert synthetic baseline is deterministic and inspectable.
- Runtime tests assert real wiring order: Garland geometry swap occurs before hierarchy swap.
- Runtime tests assert Stage 4 hierarchy uses full SAMRAS expansion path behavior (example selectable progression: `3 -> 3-2 -> 3-2-3 -> 3-2-3-17 -> 3-2-3-17-77`).
- Runtime tests assert title cells remain blank where ASCII title is missing.
- Frontend/architecture tests assert ordered renderer function exists, mode dispatch is active, and legacy renderer remains available.
- Contract docs and docs-alignment tests are updated to reflect scaffold-first reset plus explicit ordered-hierarchy mode contract.

**Assumptions and Locked Defaults**
- Delivery scope is Stages 1-4 in this cycle.
- Docs and tests are updated in the same cycle.
- Ordered hierarchy depth is dynamic (no hard cap).
- Structural expand/compress behavior is implemented now.
- Title fallback is blank-only.
- Stage 4 hierarchy target is full SAMRAS expansion, not partial emitted-node mapping.
- If full SAMRAS expansion cannot be derived from available historical/local sources, Stage 4 is treated as blocked rather than downgraded.
