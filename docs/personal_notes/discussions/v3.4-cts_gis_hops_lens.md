> I’m checking the current CTS-GIS, SAMRAS, and lens-related V2 surfaces first so the answer matches the repo shape rather than only the idea.

Yes. A lens is the correct avenue for that.

The important boundary is this: the lens should control presentation state, but it should not become the authority that decodes datum semantics on the client. Your current CTS-GIS contract already says datum authority stays with the datum-document seam and that client JS renders a server-composed projection rather than decoding coordinate overlays itself. 

That means the right shape is:

The specialized CTS-GIS mediation decodes HOPS-backed rows into a derived GeoJSON projection, attaches SAMRAS/profile information as feature properties or overlay values, and emits that in `surface_payload.map_projection`. The UI lens then chooses how to render it: auto overlay, raw only, raw underlay visible, selected document, selected row, selected feature. That is already close to your current CTS-GIS surface contract, which includes `map_projection`, `lens_state`, `overlay_mode`, and `raw_underlay_visible`.  

So the answer is not “put HOPS-to-GeoJSON in the UI module.” The answer is “put the visualization control in the lens, but keep HOPS-to-GeoJSON in the specialized mediation/service layer.”

That is also algorithmically coherent with the older maps implementation shape. In that service, HOPS coordinate tokens are classified and decoded, rows with HOPS bindings become point or polygon GeoJSON features, and title/SAMRAS/HOPS overlays are treated as lens-driven display layers rather than datum replacement. 

On caching, there is already an obvious pattern in your current CTS-GIS runtime. `build_admin_cts_gis_surface_payload` caches the derived surface payload for 30 seconds using a key composed from data root, tenant, selected document, selected row, selected feature, overlay mode, and raw-underlay visibility.  That is the right class of cache for this feature.

The next refinement should be this exact split:

Keep one projection cache at the mediation/runtime layer keyed by `document_id + overlay_mode + raw_underlay_visible`. That cache should hold the expensive derived GeoJSON and decoded overlay payload. Then keep selection as a cheap view-state operation on top of that cached projection. Right now your cache key includes selected row and selected feature, which is safe, but heavier than necessary for large documents. The geometry does not need to be recomputed just because the selected feature changed. 

For preloading, the clean path is to preload only the active document’s projection plus lightweight catalog summaries for the others. Your current surface shape already supports that because it separates `document_catalog`, `selected_document`, `map_projection`, `rows`, and `lens_state`.   The first load should therefore return:
the selected document’s full feature collection,
document summaries for sibling CTS-GIS documents,
bounds and feature counts for fast switching.

If you want the map to live in the dominant interfacing panel instead of the workbench, that is also structurally fine, but use the same `surface_payload`. Do not create a second projection pipeline. Your shell already supports tool-dominant interface patterns, and the current portal JS already renders the geographic pane from `surface_payload` rather than from an independently decoded client state.  

So the concrete recommendation is:

Use CTS-GIS specialized mediation to derive GeoJSON from HOPS rows.
Treat SAMRAS id profiles as feature metadata and overlay content, not geometry authority.
Let the lens own only display toggles and selection state.
Cache projection by document and lens mode, not by every selection.
Reuse the single `surface_payload` whether the geographic pane is shown in the workbench or moved into the dominant interface panel.

The only caution is naming drift. Your live CTS-GIS contract is clean, but compatibility evidence still mentions legacy `maps` data roots and older tool ids. Keep that compatibility only at the intake boundary and do not let the UI or new mediation surfaces fall back to `maps` as live V2 naming.
