> I’m checking the current CTS-GIS, SAMRAS, and lens-related V2 surfaces first so the answer matches the repo shape rather than only the idea.

Yes. A lens is the correct avenue for that.

The important boundary is this: the lens should control presentation state, but it should not become the authority that decodes datum semantics on the client. Your current CTS-GIS contract already says datum authority stays with the datum-document seam and that client JS renders a server-composed projection rather than decoding coordinate overlays itself. 

That means the right shape is:

The specialized CTS-GIS mediation decodes HOPS-backed rows into a derived GeoJSON projection, attaches SAMRAS/profile information as feature properties or overlay values, and emits that in `surface_payload.map_projection`. The UI lens then chooses how to render it: auto overlay, raw only, raw underlay visible, selected document, selected row, selected feature. That is already close to your current CTS-GIS surface contract, which includes `map_projection`, `lens_state`, `overlay_mode`, and `raw_underlay_visible`.  

So the answer is not “put HOPS-to-GeoJSON in the UI module.” The answer is “put the visualization control in the lens, but keep HOPS-to-GeoJSON in the specialized mediation/service layer.”

That is also algorithmically coherent with the older maps implementation shape. In that service, HOPS coordinate tokens are classified and decoded, rows with HOPS bindings become point or polygon GeoJSON features, and title/SAMRAS/HOPS overlays are treated as lens-driven display layers rather than datum replacement. 

On caching, there is already an obvious pattern in your current CTS-GIS runtime. `build_admin_cts_gis_surface_payload` caches the derived surface payload for 30 seconds using a key composed from data root, tenant, selected document, selected row, selected feature, overlay mode, and raw-underlay visibility.  That is the right class of cache for this feature.

The next refinement should be this exact split:

Keep one projection cache at the mediation/runtime layer keyed by `document_id + overlay_mode + raw_underlay_visible`. That cache should hold the expensive derived GeoJSON and decoded overlay payload. Then keep selection as a cheap view-state operation on top of that cached projection. Right now your cache key includes selected row and selected feature, which is safe, but heavier than necessary for large documents. The geometry does not need to be recomputed just because the selected feature changed. 

For preloading, the clean path is to preload only the active document’s projection plus lightweight catalog summaries for the others. Your current surface shape already supports that because it separates `document_catalog`, `selected_document`, `map_projection`, `rows`, and `lens_state`.   The first load should therefore return:
the selected document’s full feature collection,
document summaries for sibling CTS-GIS documents,
bounds and feature counts for fast switching.

If you want the map to live in the dominant interfacing panel instead of the workbench, that is also structurally fine, but use the same `surface_payload`. Do not create a second projection pipeline. Your shell already supports tool-dominant interface patterns, and the current portal JS already renders the geographic pane from `surface_payload` rather than from an independently decoded client state.  

So the concrete recommendation is:

Use CTS-GIS specialized mediation to derive GeoJSON from HOPS rows.
Treat SAMRAS id profiles as feature metadata and overlay content, not geometry authority.
Let the lens own only display toggles and selection state.
Cache projection by document and lens mode, not by every selection.
Reuse the single `surface_payload` whether the geographic pane is shown in the workbench or moved into the dominant interface panel.

The only caution is naming drift. Your live CTS-GIS contract is clean, but compatibility evidence still mentions legacy `maps` data roots and older tool ids. Keep that compatibility only at the intake boundary and do not let the UI or new mediation surfaces fall back to `maps` as live V2 naming.  

The next design artifact to write is `docs/contracts/cts_gis_hops_projection_lens_contract.md`.

## UPDATED CONSIDERTIONS

> I’m checking the current Hanus interface note and the live CTS-GIS/runtime surfaces together, then I’ll restate the recommendation against the updated repo shape rather than the earlier one.

Yes, but the repo now makes the right answer more specific than before.

With the Hanus note, the core idea is no longer just “a map lens.” It is an attention/intention mediation surface. Your note frames the interface around an attention state and an intention state rather than a simple 1:1 selected node, and the live CTS-GIS service now already mirrors that structure with `attention_document_id`, `attention_node_id`, and `intention_token`, followed by a later selection phase for row and feature focus.  

So the correct architectural answer is still: use a lens, but do not make the lens the authority for HOPS decoding or GeoJSON derivation. Under the live CTS-GIS service, HOPS tokens are classified and decoded server-side, row projections are built there, and features are emitted with SAMRAS/profile metadata attached. Geometry comes from HOPS; profile meaning comes from SAMRAS overlays and profile summaries. 

That means the clean Hanus split is now this. Attention should be the document/node anchor. Intention should be the render directive over that anchor. Selection should be a later, cheaper focus over the already-built render set. Lens state should remain presentation-only, mainly `overlay_mode` and `raw_underlay_visible`. That is exactly the direction the service is taking now: projection bundle first, mediation surface second, selection finalization third.  

Because of that, the caching answer also changes slightly from the earlier formulation. It is no longer best described only as `document_id + overlay_mode + raw_underlay_visible`. In Hanus terms, there are two expensive stages, not one. The repo now already separates them this way: a projection cache keyed by data root, tenant, overlay mode, raw-underlay visibility, and target document; then a mediation cache that adds attention document, attention node, and intention token; then `finalize_selection(...)` applies row/feature selection on top of that. That is the correct split. Selection should stay cheap and uncached at the heavy geometry stage. 

So for your question about a GeoJSON window in the dominant interfacing panel: yes, that is now the right surface for CTS-GIS. The shell contract now explicitly classifies CTS-GIS as `interface_panel_primary`, drives `foreground_shell_region` to `interface-panel`, opens the inspector by default for that posture, and issues a fallback workbench card only when the panel is collapsed.  The shell core also applies that posture on the client by marking the inspector as the primary surface and switching the shell into interface-panel foreground mode. 

So the GeoJSON window should live in the dominant interface panel as a Hanus lens view over the same canonical `surface_payload.map_projection`. It should not create a second projection pipeline in the browser. The browser should render the server-issued feature collection and dispatch attention/intention changes back through the CTS-GIS request contract. That keeps the UI module thin and keeps datum authority out of client code.  

On the SAMRAS side, the right treatment is still to use SAMRAS ids and profiles as feature metadata, lineage, child/related-profile context, and attention membership, not as geometry authority. The current service already enriches feature properties with `samras_node_id`, `profile_label`, `lineage`, `parent_node_id`, and `attention_member` while leaving geometry grounded in decoded HOPS coordinates. 

So the reconsidered recommendation is:

Use CTS-GIS specialized mediation to build a Hanus-shaped surface: projection bundle, then mediation surface, then selection finalization. Put the GeoJSON widget in the dominant interface panel. Keep HOPS-to-GeoJSON in the service layer, not in the UI lens. Let the lens own only display mode and operator focus. Cache projection by document and lens inputs, cache mediation by attention and intention, and treat row/feature selection as the cheap final pass.
