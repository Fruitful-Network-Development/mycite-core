**OUTPUT**

The repo is cleaner than you seem to think. Its strongest quality is that V2 already has an explicit authority order, explicit surface ownership, and a real separation between shell state, runtime composition, adapters, modules, and host transport. That is not accidental; it is deliberate architecture.   

Your datum handling also does not look like the main source of heaviness. The datum ref normalization is strict, and the AITAS context is already forcing qualified datum references into normalized form, which is a good sign for keeping state-machine inputs disciplined rather than letting arbitrary UI strings become authority.  

The actual heaviness is elsewhere. You are building four things at once: a shell/runtime platform, a tool-family platform, a hosted client portal product, and a future network-contract system. The repo already has a clean family-root direction for AWS-CSM, FND-EBI, FND-DCM, and PAYPAL-PPM, with read-only-first and bounded-write-later discipline.    

Where it will start to feel complicated is here.

First, registry truth is duplicated. The shell-owned registry lives in `admin_shell.py`, while runtime entrypoint truth lives in `runtime_platform.py`. That is acceptable for a few tools, but once you add host aliases, progeny, PayPal, DCM, EBI, and per-client surfaces, you will be maintaining parallel declarations for the same family in multiple places.  

Second, your audience model is still too coarse for your future product. The current model is basically `internal` versus `trusted-tenant`, and the trusted-tenant portal is intentionally narrow. That is clean today, but it is not enough for cleaners, client operators, portal owners, network peers, or desktop-app identities.  

Third, the hosted/progeny/network part of the architecture is not yet first-class in V2. The current planning index explicitly holds `tenant_progeny_profiles` out, and that held-out doc says no V2 tool exists yet and that hosted/progeny domain ownership must be defined before approval. That is the clearest sign of where the missing architecture still is.  

Fourth, your new taxonomy is not yet formalized. You now think in terms of default tools, tools, service tools, and a future host-alias tool. I do not see that taxonomy codified as a V2 contract yet. Right now the repo has family roots and tool exposure rules, but not a formal type system for these categories. The existing contract is about shell-owned legality and visibility gating, not about multi-kind tool ontology. 

My view is that the app is not too heavy because of the datum model. It is heavy because the missing boundary is between “tool surface” and “network relation.” Progeny, alias, and P2P contract anchoring should not first appear as UI tools. They need to exist as their own domain model first. Then tools can project, inspect, and edit bounded parts of that model.

The plan I would use is this.

1. Freeze a canonical V2 taxonomy for tool kinds. Define `root_service`, `family_root_tool`, `service_tool`, and `host_alias_tool`. For each kind, define who owns identity, who owns legality, what its visibility gate is, and whether it may compose subordinate families.

2. Create one canonical static catalog for tool/family declaration. Do not use dynamic scanning. The shell registry and runtime catalog should both derive from one source instead of being hand-maintained in parallel.

3. Define the missing network-domain contract before adding the host-alias surface. At minimum, you need first-class contracts for `portal_instance`, `host_alias`, `progeny_link`, `p2p_contract`, and `external_service_binding`. Those should be domain entities, not just routing concepts.

4. Expand the audience/role contract. Before you offer client portals and desktop apps, separate portal roles from network roles. “Trusted tenant” is too broad for the future system you described.

5. Make `host_alias` a family root under Utilities, not a new shell root. It should act as an orchestration family that composes FND-EBI, AWS-CSM, PAYPAL-PPM, and FND-DCM around one hosted client identity. It should not own the underlying provider truths; it should own the linkage and projection.

The most important conclusion is this: the current V2 work is structurally sound, but the next thing that must become explicit is not another surface. It is the contract layer that sits between portal instances and network relationships. Until that exists, every new family you add will keep pushing network semantics into shell and runtime code.

The next exact artifact I would create is `docs/contracts/host_alias_and_portal_instance_contract.md`.
