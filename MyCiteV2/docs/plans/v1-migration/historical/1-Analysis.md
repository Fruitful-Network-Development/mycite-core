# Structural Analysis of mycite-core Refactor and Hanus Interface Model
### Background: Tool Development and Data Handling Goals
The tool_dev.md plan describes the guidelines adopted during the refactoring of mycite‑core. These guidelines have three pillars:
    1. Separate tool code from datum authority. Canonical tool logic is placed under packages/tools/<tool> and instances/_shared/runtime/flavors, while tool‑specific datum files live under data/sandbox/<tool> and payload binaries/cache live under data/payloads/[1]. The portal’s private/config.json controls which tools are exposed but does not define the truth of any datum[2]. Non‑datum utility state goes under private/utilities/tools/<tool>[3].
    2. Maintain a strict portal surface. Tools must attach through portal‑provided mediation surfaces (e.g., workbench layout, context payloads) rather than inventing ad‑hoc logic[4]. The shell surface is emphasized over hidden convenience logic[2].
    3. Use hyphae/magnitude abstractions. Datum addresses are expressed using a nested <datum>:<reference>;<magnitude> convention (e.g., datum_a:(datum_b:(datum_c:(datum_d:0‑0‑6;magnitude_w);magnitude_x);0);magnitude_z)[5]. Families of datums are defined by a “magnitude” which determines how many bits are needed to represent the array and what lens is applied when rendering cells[6]. This allows the data engine to recognize abstraction paths and apply lenses (octet, ASCII, decimal, etc.) uniformly[7].
The plan also notes that some tools omit the third panel (the workbench) and present only a control panel and an interface panel; in such cases the interface panel is responsible for presenting custom mediations while still operating on the same NIMM–AITAS state machine logic[8]. The Maps tool example shows how the portal mediates the SAMRAS structure of administrative nodes by focusing on a facet of a datum rather than the datum itself[9]. Navigation through SAMRAS addresses (e.g., 3‑2‑3‑17‑77) leverages the NIMM/AITAS notation <mediate>[{facet}{mediate}{null}{SAMRAS}{4}]:1‑1‑2;3‑2‑3‑17‑77[10].
These guidelines ensure that tools remain modular and that their interfaces attach cleanly to the portal’s state machine rather than duplicating logic or creating drift.
### Hanus Interface Model: UI Concepts for Hierarchical Spaces
The newly added document hanus_interface_model.md proposes a generic interface model for hierarchical address spaces. The file records a conversation exploring how to visualize and interact with materialized path style IDs (e.g., 1‑1, 1‑1‑3‑1) and how to extend the concept beyond geopolitical domains (taxonomy, etc.). Key insights include:
    • The ID namespace is a materialized path where the entire lineage is embedded in the ID string[11]. This makes it efficient to derive a node’s parent or children without running recursive queries.
    • Master–detail tree and Miller columns are recommended navigation patterns. In a master–detail tree, a collapsible tree lists only the msn_id and title, while a detail pane displays the full profile for the selected node[12]. Miller columns allow deep datasets to be traversed by showing a column per depth level[13].
    • Action modals leverage the path semantics: adding a node automatically assigns an ID by appending a next available number to the parent’s ID[14].
    • The model emphasises separating navigation state from render state. A simple 1:1 relationship between the selected node and the rendered geometry traps the user when they need to view a parent’s children without changing focus or to filter nodes by type across irregular branches. To solve this, the document proposes a compound state model with three parameters:
    • Focus – the anchor ID (e.g., focusId: 3‑2 for the United States). It determines the context of the profile pane[15].
    • Resolution (Depth) – how deep relative to the focus the render engine should traverse (e.g., depth: 1 shows immediate children)[16].
    • Semantic filter – a list of types to include or exclude (e.g., only show counties when Puerto Rico’s barrios would otherwise appear at the same depth)[17].
    • The document introduces a dual‑string state approach using Attention and Intention. These strings share the same namespace syntax: Attention denotes the focal node, while Intention encodes traversal rules using 0 as a wildcard. For example, attention=3‑2 (US) and intention=3‑2‑3‑0‑0 renders counties across states by following branch 3 (county) and then two levels of wildcards[18]. Because both states are serialized strings, they can be manipulated through a button‑shell interface and bookmarked in URLs[19].
Taken together, the Hanus interface model proposes that the UI should be decoupled from specific domains. The navigation component handles ID traversal (attention/intention) and the presentation component listens for changes and renders the appropriate projection (maps, hierarchical lists, or other visualizations) based on focus, resolution and semantic filters.
### HOPS and Chronological/Spatial Datums: Structural Considerations
The HOPS document describes a Homogeneous Ordinal Partition Structure—a fixed, level‑homogeneous schema for ordered address spaces such as time or coordinates. HOPS defines only the structure (levels and capacities) and not the occupied nodes[20]. Its contrast with SAMRAS is that SAMRAS reconstructs variable tree shape, while HOPS defines uniform subdivision[21].
Important points:
    • Chronological HOPS (1‑1‑1) partitions time into denotational capacities like quadrenniums (1461 days) and hours/minutes/seconds[22]. A date is thus represented as an ordinal container path—e.g., April 6 2026 becomes 4‑447‑751‑507‑827[23].
    • Spatial HOPS (1‑1‑3) partitions space into an octet (8 parts), then a 9×9 grid (81 parts) and then repeated 10×10 grids (100 parts)[24]. The first level uses an ordered naming such as NEG, NEH etc.[25].
    • HOPS addresses are interpreted by selecting ordinal cells inside the fixed partition schema; a coordinate is converted into a HOPS address by recursive containment[26]. Conversely, an address can be projected into a GeoJSON bounding region or centroid[27].
    • The functional separation emphasises three layers: structure (HOPS magnitude), address (ordinal path) and projection (contextual meaning)[28]. The UI should operate on addresses and ask the engine to project them; it should not manipulate raw bitstreams[29].
The system log file (system_log.json) shows how HOPS and SAMRAS structures are applied to calendar datums. Entries like 7‑3‑1 and 7‑3‑4 contain arrays referencing other datums via ref.3‑1‑1 (chronological HOPS) and ref.3‑1‑4 (seemingly SAMRAS facet) along with a bitstring representing the event payload[30]. Each entry has a human‑friendly label (e.g., first_christmas_0000_01_01, y2k_1999_12_31, americas_250th_anniversary_2026_07_04) and an event type. The event’s chronological address (4‑447‑751‑507‑916 for July 4 2026) is encoded in the fifth element of the value array. This illustrates how the system uses HOPS‐based chronological addresses to timestamp events and SAMRAS structures to classify them.
## Implications for UI Shell Design
### 1. Interface Surfaces Should Be Domain‑Agnostic
The Hanus model advocates a navigation component that understands materialized paths but does not impose semantics beyond the ID pattern. A tree viewer, Miller columns or a button‑shell interface should not hard‑code the meaning of each segment; instead, they operate on attention and intention strings and rely on the engine to fetch profiles and projections. This matches the tool_dev guideline that the portal’s shell should be primary and tool logic should attach through defined surfaces[2]. By keeping the navigation state generic, the same component can handle SAMRAS structures (variable branching) and HOPS structures (uniform branching); it simply passes the ID and context to the engine.
### 2. Decouple Navigation State from Render State
HOPS demonstrates that an address can be interpreted at different specificities—shorter addresses indicate less specificity within the same structure[31]. Likewise, the Hanus compound state (focus/resolution/semantic filter) or attention/intention strings decouples where the user is exploring from what is rendered. For instance, in a calendar tool built on chronological HOPS, the focus might be a specific quadrennium (e.g., 4‑447‑751‑507), the resolution might indicate whether to show days or hours, and semantic filters could restrict event types (e.g., show only general_event). In a SAMRAS navigation, the intention string can traverse specific branches (e.g., show counties but not municipalities).
### 3. Treat Time as Space (Calendar Tool)
The user hints at a devoted calendar tool that interfaces a timeline as if it were space. Chronological HOPS provides exactly this: time partitions into ordinal containers (quadrenniums, days, hours, etc.) which can be navigated like spatial cells. A calendar UI built on the Hanus model would:
    • Use the attention string to anchor the navigation (e.g., focus on a specific year/quadrennium).
    • Use intention or resolution controls to adjust depth—e.g., show just days (intention = attention), hours (attention‑0), or all descendants.
    • Use semantic filters to include/exclude event types or only show events with certain tags.
Because the timeline is treated as an ordered space rather than an absolute date string, this UI would unify the handling of historical and future events across different scales. The system_log.json exemplifies how events are encoded using chronological addresses and event types; the calendar tool could read those addresses, ask the engine to project them into Gregorian dates or relative times, and render them in the interface.
### 4. Maintain Separation of Structural, Address and Projection Layers
Both HOPS and the Hanus model emphasize separation between structure, address and projection. The UI should never encode or decode bitstreams of HOPS or SAMRAS; it should operate with human‑friendly IDs and rely on the data engine to validate and project them[29]. For example, the map tool uses a SAMRAS structure to load geo‑coordinates via a geo coordinate HOPS address lens[32]. Similarly, the calendar tool should call the engine to project a chronological address into a Gregorian date or timestamp.
### 5. Generic UI Shell for Multiple Tools
The portal’s current UI shell can be simplified by abstracting common patterns: navigation panel, control panel, and interface panel. Tools should provide contextual mediations (e.g., map visualization, calendar view) that plug into the interface panel, while the navigation component handles ID traversal generically. This reduces situational specificity and allows the same shell to wire into different interface models (SAMRAS navigation, Hanus button‑shell, chronological HOPS timeline). Because tools are hosted by the portal and interact through defined surfaces[2], a well‑designed generic shell will make adding new tools (like the calendar) easier without duplicating navigation logic.
## Conclusion
The refactor of mycite‑core moves toward a clear authority chain for data and code and emphasizes that the portal shell is primary. The newly added Hanus interface model builds on this by proposing a generic, serializable navigation state for hierarchical address spaces. When combined with the HOPS abstraction for chronology and space, this model suggests a unified approach to UI design:
    1. Keep structural and domain concerns separate. Use structures (SAMRAS or HOPS) to define valid partitions, addresses to point into those structures, and projections to render them.
    2. Design navigation components that understand materialized path IDs and support compound state (focus/resolution/filter).
    3. Treat time like space. Build a calendar tool that navigates chronological HOPS addresses using the same attention/intention logic used for spatial or administrative hierarchies.
    4. Expose consistent portal surfaces. Tools attach via a generic shell that provides navigation and control panels, while custom visualizations plug into the interface panel.
This approach will reduce situational specificity in the UI, enable better wiring of tools to structural models, and support future expansions such as taxonomy explorers or timeline interfaces without rewriting the underlying navigation logic.
