# Ontology Terms

Use these terms exactly. See [../contracts/terminology_usage_rules.md](../contracts/terminology_usage_rules.md).

- `MyCiteV2`: the v2 scaffold in this tree. Do not shorten to `MyCite2` unless quoting a historical document title.
- `shell`: the state-owned legality surface for navigation and interaction.
- `tool`: a capability package that attaches through shell-defined surfaces.
- `module`: a bounded code owner with explicit inputs, outputs, and dependency rules.
- `domain module`: a module that owns domain semantics.
- `cross-domain module`: a module that serves multiple domains without becoming a generic utility dump.
- `port`: an inward-facing contract for external capability.
- `adapter`: an outward-facing implementation of a port.
- `sandbox`: an orchestration boundary for staged work and mediated interaction.
- `mediation_surface`: serialized shell/interface-state behavior for viewing or acting on a subject.
- `attention`: the current subject of the state machine.
- `intention`: the contextual traversal or interaction posture paired with attention.
- `AITAS`: attention, intention, time, archetype, and spatial context vocabulary used by the state machine.
- `Hanus interface surface`: the serialized button-shell style interface model that shapes shell interaction without becoming host logic.
- `NIMM`: the directive set for navigate, investigate, mediate, and manipulate.
- `HOPS`: homogeneous ordinal partition structure.
- `datum authority`: the explicit source of truth for a datum.
- `derived artifact`: any output regenerated from authoritative inputs, including payload binaries and caches.
- `runtime composition`: host wiring that composes inward layers without owning their semantics.
